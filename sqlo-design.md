## 1. Executive Summary

### What is Sqlo?

**Sqlo** is an AI-powered SQL practice platform where users create persistent datasets — either generated by AI, uploaded from CSV, or built manually — and generate unlimited practice problems on top of them.

Unlike every existing SQL practice platform (DataLemur, StrataScratch, LeetCode, SQLZoo), which use **fixed, pre-written question banks**, Sqlo generates fresh problems on demand. Every answer is unique. Nothing can be Googled.

### The Core Loop

```
Create Dataset (AI or upload)
        ↓
Generate Problems (AI writes problem + hidden solution)
        ↓
Solve Problems (write SQL in browser)
        ↓
Auto-graded (result comparison, not string matching)
        ↓
Score tracked per dataset
        ↓
Add more problems anytime
```

### The One-Sentence Pitch

> "Build a SQL database with AI, create practice problems, solve them, track your score."

### Why Now

- Data Scientist is the **4th fastest-growing job in the US** (BLS, +34% by 2034)
- People are already using **ChatGPT as an ad-hoc SQL tutor** — but with no sandbox, no validation, no tracking
- Every SQL practice platform has a **leaked-answer problem** — Reddit and GitHub are full of solution dumps
- No platform offers **persistent, user-owned datasets** with AI-generated infinite problems

## 4. Product Design

### 4.1 Core Concept: Datasets as Persistent Assets

The fundamental design principle of Sqlo is that **datasets are owned objects**, not throwaway sessions.

| Session-based (old idea) | Dataset-based (Sqlo) |
|--------------------------|----------------------|
| Generate → practice → gone | Create once → practice forever |
| New setup every time | Build up a library over time |
| Platform owns all content | User owns their datasets |
| No investment in platform | High switching cost after building datasets |
| Churn after interview prep | Stays as content library grows |

When a user spends an hour building a dataset with 20 problems, **they will never leave**. Their content is there. This is the core retention mechanic — not streaks or badges, but ownership.

### 4.2 The Three Creation Methods

Users can create datasets three ways:

**Method 1: AI Generate (primary)**
```
"Create an e-commerce database with customers, orders,
products, and categories. Make it realistic with 1,000 rows.
I want to practice GROUP BY and window functions."
        ↓
AI generates schema + faker fills data
        ↓
Dataset ready in ~5 seconds
```

**Method 2: Upload CSV**
```
User uploads their own CSV files
        ↓
System infers schema and types
        ↓
AI suggests relationships between tables
        ↓
Dataset created from real data
```

**Method 3: Manual Schema**
```
User defines tables and columns manually
        ↓
AI fills with realistic fake data
        ↓
Or user seeds their own data
```

### 4.3 Dataset Anatomy

A dataset contains:

```
Dataset: "My E-commerce Practice"
├── Metadata
│   ├── name, description, created_at
│   ├── tags: [e-commerce, GROUP BY, window functions]
│   └── visibility: private / public
│
├── Schema
│   ├── orders (order_id, customer_id, total_amount, order_date, status)
│   ├── customers (id, name, email, country, created_at)
│   ├── products (id, name, category, price, stock)
│   └── categories (id, name, parent_category)
│
├── Data
│   └── ~1,000 rows per table (SQLite file)
│
└── Problems
    ├── Problem 1: "Find top 5 customers by total spend" [solved ✅]
    ├── Problem 2: "Monthly revenue trend for 2024" [solved ✅]
    ├── Problem 3: "Revenue by category with HAVING" [unsolved ❌]
    └── Problem 4: "Running total using window function" [unsolved ❌]
```

### 4.4 Problem Anatomy

Each problem contains:

```
Problem:
├── description    → shown to user (natural language, no SQL hints)
├── concept_tags   → ['GROUP BY', 'HAVING']
├── difficulty     → easy / medium / hard
├── hint_1         → conceptual hint (free)
├── hint_2         → structural hint (free)
├── hint_3         → near-solution hint (free)
├── solution_query → NEVER sent to frontend
└── points         → 10 / 25 / 50 based on difficulty
```

### 4.5 Dataset Score

Score is simple and honest:

```
Dataset Score = points earned / total points available

E-commerce:   (10 + 25) / (10 + 25 + 25 + 50) = 35/110 = 32%
Hospital:     (10 + 10 + 25) / (10+10+25+25+50) = 45/120 = 38%

Overall Score = total points earned across all datasets
```

No XP inflation. No gamification complexity. Just honest progress.

### 4.6 Adding More Problems

At any time the user can:

```
[+ Generate Problems]

How many? [5]
Concept focus: [Window Functions ▼]
Difficulty: [Hard ▼]

AI generates 5 new problems specifically for THIS dataset's schema
→ Each problem verified: solution runs and returns results
→ Added to the dataset immediately
```

This is the key mechanic that makes datasets grow with the user. Start with 5 easy problems, add 5 hard ones when ready. The dataset evolves.

### 4.7 Dataset Visibility & Sharing

```
Private (default)   → only you can see and practice
Public              → listed in community library
                    → others can practice (read-only)
                    → original owner gets "attribution"
```

**Community datasets** are the free content engine. Power users build great datasets, share them publicly, and other users practice on them. No manual curation needed.

## 5. User Flows

### 5.1 New User Journey

```
Step 1: Sign up (email or Google OAuth)
        ↓
Step 2: Onboarding (3 questions)
        → What's your SQL level? [Beginner / Intermediate / Advanced]
        → What's your goal? [Learning / Interview prep / Work practice]
        → Preferred industry? [E-commerce / Healthcare / Finance / Other]
        ↓
Step 3: First dataset created automatically
        → Based on onboarding answers
        → 5 starter problems pre-generated
        → "Your first dataset is ready — try solving a problem"
        ↓
Step 4: Solve first problem → get score
        ↓
Step 5: Dashboard shows progress
```

> **Key principle:** Zero time between signup and first "win". User must solve a problem within 2 minutes of signing up or they will churn.

### 5.2 Core Practice Loop (Returning User)

```
Open Sqlo
        ↓
Dashboard: see all datasets with scores
        ↓
Pick a dataset (or create new one)
        ↓
Pick an unsolved problem
        ↓
Read problem → write SQL → run
        ↓
        ├── Correct → points added → next problem
        └── Wrong   → see what's different → try again
                         ↓ (still stuck)
                    Request hint (up to 3 levels)
                         ↓ (still stuck)
                    AI step-by-step solve
                    → problem marked "solved with help"
                    → no points, but learn from walkthrough
```

### 5.3 Stuck Flow (Critical Path)

This flow must teach, not just give answers:

| Stage | What Happens |
|-------|-------------|
| Wrong answer | Show row count diff: "You got 8 rows, expected 5 rows" |
| Hint 1 (free) | Conceptual: "Think about which tables have revenue data" |
| Hint 2 (free) | Structural: "You'll need a JOIN and a GROUP BY here" |
| Hint 3 (free) | Near-solution: "Start with: SELECT category, SUM(amount)..." |
| AI Walkthrough | Step-by-step thinking process, no SQL yet |
| Full Solution | Complete SQL with line-by-line explanation. No points awarded. |
| Try Again | Offer to generate a similar problem immediately |

### 5.4 Dataset Creation Flow

```
[+ New Dataset]
        ↓
Choose creation method:
◉ AI Generate  ○ Upload CSV  ○ Manual
        ↓
AI Generate path:
→ Type prompt: "Hospital database with doctors, patients,
   appointments. Practice JOINs and subqueries."
→ Choose row count: [Small ~100] [Medium ~1K] [Large ~10K]
→ [Generate Dataset]
        ↓
~5 seconds
        ↓
Preview:
→ Schema diagram shown
→ Sample rows shown
→ [Looks good — Generate Problems] or [Regenerate]
        ↓
Problem generation:
→ How many problems? [5 / 10 / Custom]
→ Concept focus? [Mixed / Specific concept]
→ Difficulty? [Mixed / Easy / Medium / Hard]
→ [Generate Problems]
        ↓
~3 seconds
        ↓
Dataset ready — start practicing
```

### 5.5 Teacher / Classroom Flow

```
Teacher creates account → upgrades to Teacher plan
        ↓
Creates cohort → gets invite link
        ↓
Builds dataset (or uses AI) → adds problems
        ↓
Assigns dataset to cohort with due date
        ↓
Students join via link → practice assigned problems
        ↓
Teacher views class dashboard:
→ Each student's score per problem
→ Who is stuck, who is done
→ Common wrong answers (aggregate view)
        ↓
Teacher can add more problems mid-assignment
```

## 6. System Architecture

### 6.1 High-Level Overview

```
┌─────────────────────────────────────────────────────┐
│                  BROWSER (Next.js)                  │
│   Pages | SQL Editor | Dataset Viewer | Dashboard   │
└──────────────────┬──────────────────────────────────┘
                   │ HTTPS + JWT
┌──────────────────▼──────────────────────────────────┐
│               BACKEND (FastAPI)                     │
│  /datasets  /problems  /execute  /evaluate          │
│  Auth middleware | Rate limiting | Query sandbox    │
└──────────┬─────────────────────┬────────────────────┘
           │                     │
┌──────────▼──────────┐ ┌────────▼──────────────────┐
│  Supabase           │ │  OpenAI API               │
│  PostgreSQL         │ │  Schema gen               │
│  Auth + Storage     │ │  Problem gen              │
└──────────┬──────────┘ │  Evaluation               │
           │            └───────────────────────────┘
┌──────────▼──────────┐
│  SQLite Files        │
│  (per dataset)       │
│  Practice databases  │
└─────────────────────┘
```

### 6.2 Architecture Decisions

| Decision | Rationale |
|----------|-----------|
| **SQLite per dataset** | Each dataset = isolated SQLite file. Prevents cross-user access. Fast for small datasets. No shared state. |
| **Supabase for app data** | User accounts, datasets metadata, problems, attempts, scores. Real PostgreSQL + auth built in. |
| **FastAPI backend** | Python ecosystem for AI/data — pandas, faker, SQLAlchemy. Async support for concurrent users. |
| **Next.js frontend** | Server components for SEO (landing page). Monaco Editor for SQL. React for interactive UI. |
| **OpenAI GPT-4o** | Schema generation, problem writing, evaluation. Called sparingly to control costs. |

### 6.3 Request Flow — Dataset Creation

```
POST /datasets/create
body: { prompt: "e-commerce...", row_count: 1000 }

1. FastAPI → OpenAI: "Generate normalized e-commerce schema"
   Output: JSON schema (tables, columns, relationships)

2. FastAPI → Python faker + pandas:
   Generate realistic rows matching schema
   (NOT AI — much faster and cheaper)

3. FastAPI → Create SQLite file:
   Path: /data/datasets/{user_id}/{dataset_id}.db
   Insert all rows

4. FastAPI → Supabase:
   Store dataset record + schema JSON

5. Return to frontend:
   { dataset_id, schema, sample_rows }
```

### 6.4 Request Flow — Query Execution

```
POST /datasets/{id}/execute
body: { problem_id, query: "SELECT..." }

1. Validate query:
   - Only SELECT allowed
   - No DROP/DELETE/INSERT/UPDATE/ALTER
   - Timeout: 5 seconds

2. Execute user query against dataset SQLite

3. Execute stored solution against same SQLite

4. Compare:
   is_correct = sorted(user_result) == sorted(solution_result)
   (SET comparison, never string comparison)

5. If correct:
   - Award points
   - Update dataset score
   - AI generates short optimization tip

6. If wrong:
   - Record attempt
   - Return: row count diff + sample expected vs actual

7. Store attempt in Supabase
```

## 7. Database Schema

### 7.1 Users & Profiles

```sql
-- Managed by Supabase Auth (auth.users)
CREATE TABLE profiles (
  id            uuid PRIMARY KEY REFERENCES auth.users(id),
  username      text UNIQUE,
  display_name  text,
  avatar_url    text,
  level         text DEFAULT 'beginner', -- beginner/intermediate/advanced
  goal          text,                    -- 'learn'/'interview'/'work'
  industry_pref text,
  plan          text DEFAULT 'free',     -- free/pro/teacher
  total_score   integer DEFAULT 0,
  created_at    timestamp DEFAULT now()
);
```

### 7.2 Datasets

```sql
CREATE TABLE datasets (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       uuid REFERENCES auth.users(id),
  name          text NOT NULL,
  description   text,
  schema_json   jsonb NOT NULL,          -- full schema definition
  db_path       text NOT NULL,           -- path to SQLite file
  row_counts    jsonb,                   -- { "orders": 1000, "customers": 250 }
  tags          text[],                  -- ['e-commerce', 'GROUP BY']
  visibility    text DEFAULT 'private',  -- 'private' / 'public'
  total_points  integer DEFAULT 0,       -- sum of all problem points
  earned_points integer DEFAULT 0,       -- points user has earned
  created_at    timestamp DEFAULT now(),
  updated_at    timestamp DEFAULT now()
);

-- Score as computed percentage
-- score = earned_points / NULLIF(total_points, 0) * 100
```

### 7.3 Problems

```sql
CREATE TABLE problems (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  dataset_id     uuid REFERENCES datasets(id) ON DELETE CASCADE,
  order_index    integer NOT NULL,
  description    text NOT NULL,          -- shown to user
  concept_tags   text[],                 -- ['GROUP BY', 'HAVING']
  difficulty     text NOT NULL,          -- 'easy'/'medium'/'hard'
  points         integer NOT NULL,       -- 10 / 25 / 50
  hint_1         text,
  hint_2         text,
  hint_3         text,
  solution_query text NOT NULL,          -- NEVER sent to frontend
  order_matters  boolean DEFAULT false,
  created_at     timestamp DEFAULT now()
);
```

### 7.4 Attempts

```sql
CREATE TABLE attempts (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id          uuid REFERENCES auth.users(id),
  problem_id       uuid REFERENCES problems(id),
  dataset_id       uuid REFERENCES datasets(id),
  query_written    text NOT NULL,
  is_correct       boolean NOT NULL,
  viewed_solution  boolean DEFAULT false,
  hints_used       integer DEFAULT 0,
  time_taken_secs  integer,
  points_awarded   integer DEFAULT 0,
  ai_feedback      text,
  created_at       timestamp DEFAULT now()
);

-- Track best attempt per user per problem
CREATE UNIQUE INDEX attempts_best
  ON attempts (user_id, problem_id)
  WHERE is_correct = true;
```

### 7.5 Classroom Features

```sql
CREATE TABLE cohorts (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  teacher_id   uuid REFERENCES auth.users(id),
  name         text NOT NULL,
  invite_code  text UNIQUE DEFAULT substring(md5(random()::text), 1, 8),
  created_at   timestamp DEFAULT now()
);

CREATE TABLE cohort_members (
  cohort_id  uuid REFERENCES cohorts(id),
  user_id    uuid REFERENCES auth.users(id),
  joined_at  timestamp DEFAULT now(),
  PRIMARY KEY (cohort_id, user_id)
);

CREATE TABLE assignments (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  cohort_id   uuid REFERENCES cohorts(id),
  dataset_id  uuid REFERENCES datasets(id),
  due_date    timestamp,
  created_at  timestamp DEFAULT now()
);
```

### 7.6 Community

```sql
-- When a public dataset is forked by another user
CREATE TABLE dataset_forks (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  original_id    uuid REFERENCES datasets(id),
  fork_id        uuid REFERENCES datasets(id),
  forked_by      uuid REFERENCES auth.users(id),
  created_at     timestamp DEFAULT now()
);

-- Stars/saves for public datasets
CREATE TABLE dataset_stars (
  dataset_id  uuid REFERENCES datasets(id),
  user_id     uuid REFERENCES auth.users(id),
  created_at  timestamp DEFAULT now(),
  PRIMARY KEY (dataset_id, user_id)
);
```

### 10.4 Validation Step (Critical)

After AI generates problems, **every solution is executed** against the dataset before saving:

```python
async def validate_and_save_problems(problems: list, db_path: str):
    valid = []
    for problem in problems:
        try:
            result = execute_query(problem['solution_query'], db_path)
            if len(result) >= 1:  # solution returns at least 1 row
                valid.append(problem)
            # else: discard silently, regenerate if needed
        except Exception:
            pass  # discard broken solution
    return valid
```

This is the most important engineering step. A ~10% rate of broken problems destroys user trust instantly.

## 11. Scoring System

### 11.1 Points Per Problem

| Difficulty | Points | Typical concepts |
|------------|--------|-----------------|
| Easy | 10 pts | Simple SELECT, basic WHERE, single table |
| Medium | 25 pts | JOINs, GROUP BY, HAVING, subqueries |
| Hard | 50 pts | Window functions, CTEs, complex multi-table |

### 11.2 Dataset Score

```
Dataset Score % = earned_points / total_points × 100

A dataset with 2 easy (20pts) + 2 medium (50pts) + 2 hard (100pts)
Total = 170 pts

If user solved 2 easy + 1 medium correctly:
Earned = 20 + 25 = 45 pts
Score = 45/170 = 26%
```

### 11.3 Rules

- **Solving with hints:** Full points (hints are free)
- **Viewing AI solution:** 0 points, problem marked "solved with help"
- **Multiple attempts:** Points awarded on first correct attempt only
- **Re-solving:** If already solved, no additional points
- **Dataset regeneration:** Scores reset (new data = new problems)

## 14. Security

### 14.1 Query Sandboxing

User SQL is the biggest attack surface. Every query goes through:

```python
def safe_execute(query: str, db_path: str, user_id: str) -> list:
    # 1. Only SELECT allowed
    normalized = query.strip().upper()
    if not normalized.startswith('SELECT'):
        raise HTTPException(400, "Only SELECT queries allowed")

    # 2. Block dangerous keywords
    forbidden = ['DROP','DELETE','INSERT','UPDATE','ALTER',
                 'CREATE','ATTACH','DETACH','PRAGMA','VACUUM']
    for word in forbidden:
        if re.search(rf'\b{word}\b', normalized):
            raise HTTPException(400, f"Keyword not allowed: {word}")

    # 3. Verify file belongs to this user
    expected = f"/data/datasets/{user_id}/"
    if not db_path.startswith(expected):
        raise HTTPException(403, "Access denied")

    # 4. Execute with timeout + row limit
    conn = sqlite3.connect(db_path, timeout=5.0)
    conn.execute("PRAGMA query_timeout = 5000")
    try:
        rows = conn.execute(query).fetchmany(1000)
        return [dict(row) for row in rows]
    finally:
        conn.close()
```

### 14.2 Auth & Data Isolation

- All API requests require valid Supabase JWT header
- `user_id` extracted from JWT only — never trusted from request body
- SQLite files namespaced by `user_id/{dataset_id}.db`
- Supabase Row Level Security (RLS) on all tables
- Solution queries never returned to frontend — server-side only
